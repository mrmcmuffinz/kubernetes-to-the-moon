# Container Images -- Assignment 2 of 2

This assignment is the second in a two-part series on Kubernetes container images. Assignment 1 covered Dockerfile authoring, the nerdctl build workflow, and basic image inspection. This assignment goes deeper: you will learn how multi-stage builds dramatically reduce image sizes by separating build toolchains from runtime artifacts, how Docker's layer caching works and how to order Dockerfile instructions to exploit it, how to choose an appropriate base image for different workload types, and how the OCI image format uses content-addressable storage to guarantee immutability. The capstone environment skill is running a local container registry with nerdctl and configuring a kind cluster to pull from it, which mirrors the production-realistic registry workflow you will encounter on the CKA exam and in real cluster operations. After completing this assignment, you are ready for the supply chain security topics in assignment 23 and the OPA/Gatekeeper admission policies in assignment 25, both of which build directly on the registry and image-format knowledge developed here.

## Recommended Workflow

Read through the tutorial file before attempting any exercises. The tutorial is organized as a single coherent workflow: you set up a local registry and kind cluster first, then build a Go HTTP server through several Dockerfile iterations that demonstrate multi-stage builds and layer caching, then explore base image tradeoffs, and finally push to the registry and pin an image by digest in a pod spec. Every exercise depends on the local registry being configured and running, so completing the tutorial setup section is not optional.

Work through exercises in level order. Level 1 and Level 2 exercises isolate and combine the concrete mechanics from the tutorial. Levels 3 and 5 are debugging exercises where the learning comes from reading error output and forming a hypothesis before applying a fix. The answer key for debugging exercises documents the full diagnostic reasoning, not just the corrected YAML, and reading it alongside your own investigation is where the deepest exam preparation happens. Level 4 exercises are production-style build and deploy tasks that require the registry and cluster to be working correctly, so treat any failure at Level 4 as a diagnostic exercise in its own right.

## Difficulty Progression

Level 1 exercises build basic fluency with the new mechanics from the tutorial. You will tag an image and push it to the local registry, convert a single-stage Dockerfile to multi-stage by adding a second FROM instruction, and modify a Dockerfile to demonstrate which instructions create filesystem layers versus which produce only metadata. Each exercise isolates a single concept and produces a clear, verifiable outcome within about five minutes.

Level 2 exercises combine concepts. You will reorder a poorly structured Dockerfile to maximize layer cache reuse and verify it by running two consecutive builds with only a source change, select an appropriate base image for a given workload description and justify your choice, and pin a running pod to a specific image digest rather than a mutable tag. These exercises take ten to fifteen minutes and require reasoning about tradeoffs rather than simply applying mechanics.

Level 3 exercises are debugging scenarios with bare headings. Each presents a realistic failure mode: a multi-stage Dockerfile where a COPY instruction references a stage name that does not exist, a pod spec referencing an image digest that has been corrupted, and a registry push that fails because the image tag is not formatted with the registry hostname prefix. Budget ten to fifteen minutes per exercise and read all error output before attempting a fix.

Level 4 exercises are production-style build-and-deploy tasks. You will build a multi-stage Go HTTP server image, push it to the local registry, deploy it as a Kubernetes Deployment, and verify it serves HTTP. A second exercise updates the server to a new version and completes a rolling update. A third exercise builds a Python Flask application with a Dockerfile optimized for layer caching. These exercises take fifteen to twenty minutes each.

Level 5 exercises are advanced debugging scenarios with multi-symptom failures that mirror real production incidents. The problems involve subtle interactions between build toolchain choices and runtime image constraints, registry configuration mismatches between the build host and the kind cluster, and missing runtime dependencies in minimal base images. Budget twenty to twenty-five minutes per exercise and approach each by reading logs and exit codes before touching any configuration.

## Prerequisites

This assignment assumes you have completed container-images/assignment-1. You should be comfortable with the nerdctl build and inspect workflow, Dockerfile syntax, and the nerdctl images output format from that assignment. You also need kubectl installed and comfortable familiarity with Pod and Deployment specs from the pods series. The base single-node kind cluster setup is documented in [docs/cluster-setup.md](../../../docs/cluster-setup.md#single-node-kind-cluster). Those additional steps are documented in the tutorial file itself and must be completed before the exercises.

## Cluster Requirements

This assignment uses a single-node kind cluster extended with a local container registry running on the host. The base cluster configuration is documented in [docs/cluster-setup.md](../../../docs/cluster-setup.md#single-node-kind-cluster). The tutorial file in this directory contains the complete additional setup: starting the registry container, writing the kind cluster configuration with the containerdConfigPatches block that teaches kind nodes to pull from the local registry, creating the cluster, and connecting the registry to the kind network. Follow the tutorial's environment setup section in full before starting the homework exercises. The registry must be running and the cluster must be configured to reach it for all exercises to work correctly.

## Estimated Time Commitment

The environment setup in the tutorial (registry, kind cluster with containerdConfigPatches, verification) takes fifteen to twenty minutes the first time. Reading through and running the rest of the tutorial takes an additional thirty to forty minutes. Level 1 exercises take roughly five minutes each once the environment is ready. Level 2 exercises take ten to fifteen minutes each, primarily because the Dockerfile reordering exercise requires running multiple consecutive builds to observe cache behavior. Level 3 debugging exercises take ten to fifteen minutes per exercise when you approach them by reading error output and forming a hypothesis before trying a fix. Level 4 exercises take fifteen to twenty minutes each. Level 5 exercises take twenty to twenty-five minutes each and require sustained diagnostic focus. The complete assignment, including the tutorial, runs approximately four to five hours.

## Scope Boundary and What Comes Next

This assignment deliberately excludes image vulnerability scanning, Cosign image signing, and the ImagePolicyWebhook admission controller. These topics depend on the registry workflow this assignment establishes and are covered in assignment 23 (supply chain security). OPA/Gatekeeper policies that enforce registry allowlists or prohibit mutable tags are covered in assignment 25. Dockerfile security hardening beyond the USER directive (avoiding secrets in layers, privilege escalation, read-only root filesystems) is also covered in assignment 23. This assignment uses an insecure HTTP registry to simplify the local setup. Production registries use TLS, which assignment 23 addresses in the context of Cosign signing and policy enforcement. The imagePullPolicy field and its interaction with tag conventions is covered briefly here but examined more deeply in the troubleshooting series.

## Key Takeaways After Completing This Assignment

After completing this assignment, you should be able to write a multi-stage Dockerfile that produces a final image under 10 MB for a compiled Go application and explain exactly why the size reduction occurs in terms of which filesystem layers are included in each stage. You should be able to redesign a poorly ordered Dockerfile to exploit layer caching, verify the improvement by observing CACHED labels in nerdctl build output, and articulate why the ordering matters. You should be able to select an appropriate base image for a given workload type (ubuntu, alpine, slim variant, distroless, or scratch) and explain the tradeoff between debuggability and attack surface for each choice. You should understand how OCI image manifests, configs, and layers fit together and how content-addressing via SHA256 digests makes images immutable at the format level. You should be able to push an image to a local registry, retrieve its digest, and reference that digest in a pod spec so the deployment is immune to tag mutation. You should be able to configure a kind cluster to pull from a local insecure registry using containerdConfigPatches and diagnose the ImagePullBackOff failure mode when that configuration is absent. Finally, you should be able to diagnose a container crash caused by missing runtime dependencies in a minimal base image by reading kubectl logs and container exit codes, and apply the correct fix by adjusting CGO build flags or choosing a base image that includes the required runtime.
