### DNS_2: a caught-too-late typo

**What happened:** a single missing segment (`.svc`), and you noticed it yourself, but only after submitting. This is functionally identical to the Session 1 finding on Question 6, where `/Volumes/data` versus `/Volumes/Data` cost a point on a string that was otherwise correct in approach. Session 1's plan treated that as a typo-class error requiring a mechanical fix, not an attentional one: a mandatory read-back of any exact string against the question text before moving on, not a reminder to "be more careful." The same fix applies here. The specific failure to generalize from is that the standard internal-Service FQDN form is `SERVICE.NAMESPACE.svc.cluster.local`, four dot-separated segments, not three, and `department.lima-workload.cluster.local` is missing the third. Drilling the four-segment form until it's typed automatically, and adding a one-second segment-count check before submitting any FQDN answer, closes this category of error specifically, not just this one instance of it.

**Why this is worse than it looks at 2 points.** A caught-too-late error is psychologically distinct from a simple miss, because the moment of catching it tells you the knowledge was there and the execution wasn't. That's useful information: it means more documentation reading does nothing for DNS_2, the fix is purely mechanical, a verification step inserted before submission rather than after.

**Speed target:** no new speed target needed beyond what DNS work already carries; the fix is a verification habit, not a slower build. Closed when two consecutive timed reps include an explicit segment-count or read-back check before moving to the next subtask, not just when the FQDN syntax is known.

### DNS_3: a deliberate fallback submission under uncertainty

**What happened, by your account:** you didn't know the correct mechanism for "a Pod's DNS name that survives the Pod's IP changing," so you submitted the pod-IP-based FQDN form (`IP-WITH-DASHES.NAMESPACE.pod.cluster.local`) as a known-wrong-but-something answer rather than leaving it blank. This is a genuinely different failure mode from Session 1's three outright skips (Q13, Q15, Q16), where nothing was submitted at all, and it's also different from this session's time-cascade pattern, where the cause was clock exhaustion rather than uncertainty. Here you had time, recognized the uncertainty, and made a considered choice to submit a plausible-looking wrong answer rather than nothing.

**The actual mechanism, confirmed against the official solution.** The requirement "resolves even if the Pod IP changes" is the tell that rules out any IP-based FQDN form, since an IP-based record is tied to the IP itself by definition and breaks the moment that IP changes. The correct mechanism is a Pod specifying `hostname` and `subdomain` fields in its spec, where `subdomain` matches an existing headless Service's name; that combination produces a stable DNS record of the form `HOSTNAME.SUBDOMAIN.NAMESPACE.svc.cluster.local`, which keeps resolving correctly even as the underlying Pod IP changes, because the record is keyed to the hostname/subdomain pairing rather than to an address. For this question, that's `section100.section.lima-workload.svc.cluster.local`. This is the same underlying DNS mechanism Question 11 in the Session 1 exercise bank touched on as a "pod with stable hostname" pattern; if that connection wasn't made in the moment, it's worth flagging as the specific recall gap to close, since the building block existed somewhere in prior study and didn't surface under pressure.

**Worth deciding deliberately, not by default, going forward: is a plausible-wrong fallback ever better than leaving it blank on the real exam?** On a partial-credit subtask system, a wrong answer and a blank answer score the same, zero, so there's no downside to guessing if there's truly nothing else to do with the time. The actual risk is upstream of that decision: time spent constructing a plausible-but-wrong fallback is time not spent on a different question that might have scored fully. The discipline question isn't "should I guess," it's "how long should I spend perfecting a guess before moving on," and the answer at a 90%+ target is close to zero, since a wrong guess and a fast wrong guess score identically.

**The conceptual fix.** Read the kubernetes.io/docs page on DNS for Services and Pods, specifically the section on Pod hostname and subdomain fields and how they interact with a headless Service to produce stable per-Pod DNS records. Build a Pod from scratch with `hostname` and `subdomain` set, confirm it resolves via `nslookup` from inside another Pod, then delete and recreate it with the same name to confirm the DNS record's behavior is genuinely independent of the Pod's IP, not just coincidentally stable. Speed target: under 4 minutes once the pattern is drilled, since it is a small, self-contained YAML addition once known.

### Why Q1 belongs ahead of the cascade work, not folded into it

Both DNS_2 and DNS_3 are fast, cheap fixes relative to their diagnostic value: a read-back habit and one Kubernetes DNS mechanism. Closing Q1 fully would have moved this question from 2/5 to 5/5, a three-point gain, for less total drilling time than any single time-cascade topic below requires. ### Q6, Fix Kubelet debugging (full marks, missed-the-visible-answer flag, corrected this revision)

**What actually happened, confirmed by reproducing the break and capturing the exact output.** `systemctl status kubelet --no-pager` was run, and it returned this:

```
● kubelet.service - kubelet: The Kubernetes Node Agent
     Active: activating (auto-restart) (Result: exit-code) since Sat 2026-06-27 00:36:17 UTC; 6s ago
    Process: 15460 ExecStart=/usr/share/bin/kubelet $KUBELET_KUBECONFIG_ARGS ... (code=exited, status=203/EXEC)
   Main PID: 15460 (code=exited, status=203/EXEC)
```

The complete answer is printed in that output: `status=203/EXEC` means systemd could not execute the binary, and the broken path, `/usr/share/bin/kubelet` (not a real kubelet install location, the genuine binary lives at `/usr/bin/kubelet`), is sitting in plain text on the `ExecStart=` line of the same block. This was not noticed. Instead, a guess was made that the problem was "something in the service config file" without identifying what, and from there the path to an actual fix ran through `journalctl` and then the cross-cluster comparison covered in the prior revision, a much longer route to a destination the first command had already named.

**Why this is a different and more significant finding than either prior draft of this entry.** The previous revision treated this as "the right method, one unnecessary extra step," which assumed the diagnostic information was extracted correctly and the only issue was redundant verification. That assumption was wrong. The actual gap is at the reading stage, not the tool-selection or step-count stage: the right command was run, and its output was not parsed closely enough to find the answer already there. This is closer in shape to the typo-class and read-back issues tracked elsewhere in this plan (Q1's DNS_2, Session 1's Q6 path-casing miss) than to a speed or crutch-dependency issue: it's not that a tool was missing or a step was extra, it's that the output of a correctly-chosen tool wasn't actually read for what it said.

**The specific pattern to drill: `status=203/EXEC` plus the path on the same line.** Systemd's exit-code annotations carry direct diagnostic meaning, and `203/EXEC` specifically and only means "the path in `ExecStart=` could not be executed," which collapses the entire investigation to "is that path correct" the moment the code is recognized. The path itself is printed on the `Process:` line of the same `systemctl status` block, no `journalctl`, no second cluster, and no further command needed to find it. The skill to build is reading the full `Process:`/`Main PID:` block on sight rather than skimming past it to the `Active:` line and stopping there, since `Active: activating (auto-restart)` alone tells you the service is failing but not why, and the why is one line further down.

**Remediation:** deliberately break a kubelet's systemd drop-in on a scratch cluster (the kind cluster or the QEMU single-node build, not the Pi cluster) by pointing `ExecStart=` at a wrong path, exactly as reproduced above. Run only `systemctl status kubelet --no-pager` and, before doing anything else, state out loud or write down the exit code and the exact path shown on the `Process:` line. Confirm the path is wrong by checking `whereis kubelet` or equivalent, then fix it. Repeat with two or three different injected faults (a bad flag, a missing file referenced by an `EnvironmentFile=` line, a permissions issue) to build the habit of reading the full status block rather than the habit of recognizing this one specific bug. Speed target: under 3 minutes from "node not ready" to identified root cause, since the entire diagnosis is one command, fully read, with no further investigation needed once the reading habit is in place.

### Q3, kubelet certificates (2/4): the most expensive miss on this exam, confirmed exactly as originally diagnosed

**What happened, in your words:** uncertainty about which certificate to inspect for the server-side check, time lost trying to get the `openssl x509 -ext` flag syntax right, and ultimately looking up the literal value `extendedKeyUsage` outside the allowed doc set under frustration. The submission still landed only 2 of 4 subtasks (client cert correct, server cert issuer and EKU both wrong), meaning the time spent did not even resolve the underlying confusion about which cert to check.

**Why this is the highest-priority single fix among the speed/robustness items.** This question cost time on three fronts at once (conceptual uncertainty, syntax recall, and an out-of-bounds lookup under frustration) and still came back wrong. That combination, expensive and incorrect, is worse than either an expensive-but-correct or a fast-but-wrong outcome, and it's very likely the single largest contributor to the cascade, since a 9-to-10 minute detour into the `openssl x509 -ext` discovery directly displaces an attempt at six other questions.

**The conceptual fix, confirmed by the official solution.** A kubelet has two distinct certificates, the client certificate it presents to the API server (issued by the cluster CA, `Issuer: CN = kubernetes`, Extended Key Usage = client auth) and the server certificate it presents on its own kubelet API port for things like `kubectl exec`/`logs` to connect to (self-generated on the worker node itself, so the Issuer carries the node's own hostname plus a timestamp rather than `kubernetes`, Extended Key Usage = server auth). The official solution locates these at `/var/lib/kubelet/pki/kubelet-client-current.pem` (client) and `/var/lib/kubelet/pki/kubelet.crt` (server). The score feedback's description of the submitted file (client-cert data duplicated into both slots, server cert fields showing the wrong issuer and no extensions) is consistent with having checked the same file twice rather than locating the second, distinct file.

**The syntax fix.** `openssl x509 -in <cert> -noout -ext extendedKeyUsage` is the command. There is no excuse needed for not having this cold, it just needs to be drilled until it's typed without hesitation, the same way `-enddate` for cert expiry already is from Session 1's Q14. Practice reading both Issuer (`-issuer`) and Extended Key Usage (`-ext extendedKeyUsage`) off both a client cert and a server cert on a real cluster until distinguishing them is immediate.

**The discipline fix, separate from the technical one.** Looking up an exact flag value outside the allowed doc set under frustration is a process failure independent of the certificate confusion, and it's worth naming directly: the allowed set during the exam is kubernetes.io/docs, kubernetes.io/blog, helm.sh/docs, and gateway-api.sigs.k8s.io, full stop, and `openssl` flag syntax isn't covered by any of them, meaning the real-exam version of this moment has no escape hatch at all. The fix is having `-ext extendedKeyUsage` and `-issuer` memorized before exam day so the situation that produced the frustration doesn't recur.

**Speed target:** under 6 minutes for both certs, issuer and EKU, once the two distinct cert paths and the `-ext` syntax are both cold. Closed when two clean consecutive timed reps land under target, and when a redo correctly identifies which cert is which without hesitation.

## Notes for Future Sessions

- **A zero score and a skip are not the same finding, and a zero score has at least two different causes that demand different fixes.** Session 1's zeros were confidence-based skips with time on the clock. Session B's zeros are mostly a time cascade from earlier questions, plus one genuine coverage gap.
- **A non-zero partial score can also hide two unrelated failure modes inside the same question, as Q1 shows.** DNS_2 and DNS_3 lost points for completely different reasons (a self-caught typo found too late, versus a deliberate fallback submitted under known uncertainty), and treating "Q1, 2/5" as one undifferentiated speed issue, as the first draft of this plan did, would have produced generic remediation that addressed neither actual cause.
- **A full-marks question can hide a missed-the-visible-answer error, distinct from every other category tracked so far.** Q6's reproduced failure shows the complete diagnosis printed in the first `systemctl status` output: exit code `203/EXEC` and the broken path on the `Process:` line. That output was not read closely enough, a guess was made instead, and the eventual fix came through a longer cross-cluster comparison. This is not a tool gap, not a knowledge gap, and not a speed gap in the usual sense; it's a reading-discipline gap, closest in kind to a typo-class miss but operating on command output rather than a submitted answer string. The fix is the same family of fix as the read-back habit elsewhere in this plan: read the whole output before forming a hypothesis, not just the line that looks most informative at a glance.
- **An expensive-but-wrong answer is worse signal than an expensive-but-right one, and worse than a fast skip.** Q3 cost real time and still scored 2/4. That combination is the one to watch for specifically on Session 3, since it's the failure mode most likely to cause a repeat cascade.
- **A deliberate fallback submission under known uncertainty (DNS_3's pattern) is worth distinguishing from both a guess made in ignorance and a skip made from confidence.** It is a third thing: knowing you don't know, and choosing to submit something plausible anyway rather than nothing. That choice is not wrong on a scored exam, since wrong and blank score the same, but the time spent constructing the plausible-wrong answer is the thing to watch, since that time has the same displacing effect on later questions as any other detour.
- **`kubectl explain` mid-question on an otherwise-correct answer is still a tracked flag**, the same way redirect order and hand-written RBAC were tracked in Session 1. Q6 is a different flag entirely, worth keeping separate in any future audit: not a missing skill substituted with a tool, but a correct tool's output read incompletely.
- **The two repeat-zero topics (Q15's event/log scripting, Q16's namespace introspection) are worth extra attention specifically because they repeated.** Both were skips in Session 1 and zeros in Session B, though for different proximate reasons (confidence then, time cascade now). Either way, neither topic has ever been attempted under exam conditions.
- **Q6 was corrected twice, and the corrections moved in opposite directions, which is itself worth noting.** The first draft treated it as a fragile crutch (cross-cluster comparison substituting for real skill). The second draft swung the other way, treating the method as essentially sound with one excess step. The reproduced output settled it: the method's first command was exactly right and its output was sufficient on its own, but that output wasn't read fully, which is a different and more specific finding than either earlier draft landed on. The general lesson is that a remediation entry built from memory alone, without the actual artifact (command output, submitted YAML, score feedback) in hand, can drift in either direction, too harsh or too lenient, until the artifact itself is checked. Where reproducing the failure is possible, as it was here, it's worth doing before locking in a diagnosis.
- **Documentation-based remediation drafting is reliable but not infallible, and the official solution is the thing to check before locking in a speed target, not after.** Q12's original entry scoped in more complexity than the actual exam question needed.
